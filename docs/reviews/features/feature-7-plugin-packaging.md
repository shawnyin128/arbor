# Feature 7 Review: Arbor Plugin Packaging

## Purpose

Package the accepted standalone `arbor` skill into a repo-local Codex plugin named `arbor`, without adding semantic dispatch or changing hook behavior.

## Scope

In scope:

- Create `plugins/arbor`.
- Add `.codex-plugin/plugin.json` with concrete Arbor metadata.
- Ship the accepted skill under `plugins/arbor/skills/arbor`.
- Ship the three-hook Arbor contract under `plugins/arbor/hooks.json`.
- Add tests that detect package drift between source skill, packaged skill, and canonical hook registration definitions.

Out of scope:

- Marketplace registration.
- Home-local plugin installation.
- Runtime natural-language dispatcher.
- Stage B trigger evaluation harness.
- MCP servers, apps, auth flows, or external services.
- Changes to Hook 1, Hook 2, or Hook 3 behavior.

## Design Notes

Feature 7 intentionally separates plugin packaging from trigger evaluation:

- Plugin packaging proves Arbor can be distributed as a plugin payload.
- Stage B later proves whether hook dispatch semantics work through a structured sidecar and eval harness.

The plugin currently duplicates `skills/arbor` into `plugins/arbor/skills/arbor`. Tests enforce exact synchronization so any future source change must update the package copy. A later packaging refactor can choose a single source-of-truth layout if duplication becomes costly.
